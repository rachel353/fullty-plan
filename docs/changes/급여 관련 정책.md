# 급여 관련 정책

## **1) 정책 목표 / 원칙**

### **목표**

- **실제 연봉(실제비용)은 시스템 어디에도 노출하지 않는다.**
- 모든 원가/수익성/투입비용 계산은 **“레벨 비용(=레벨 평균연봉 기반 시간당 비용)”**만 사용한다.

### **원칙**

- 직원 개인의 급여 입력/보관이 필요 없도록 설계(= **원천적으로 실제비용 데이터가 없음**).
- 권한이 높은 사용자라도 “실제 연봉”은 볼 수 없고, **레벨·레벨단가만** 본다.
- 프로젝트의 원가/마진은 “레벨단가 × 실제 투입시간(또는 예정시간)”으로 계산한다.

---

## **2) Settings > 급여별 레벨(레벨 비용) 설정**

### **2-1. 레벨 체계 개념**

- 레벨은 **지역 그룹(예: KOR / SEA) 별로 독립적으로 관리**된다.
- 예: KOR의 L1, L2와 SEA의 L1, L2는 **서로 다른 단가**를 가질 수 있다.

### **2-2. 레벨 엔티티(저장 필드)**

레벨(GradeLevel)에는 아래가 필요:

- regionGroup: KOR | SEA (추후 확장 가능)
- levelName: 예) KOR-L1, SEA-Mid 등 (자유 입력)
- avgAnnualSalary: 평균 연봉(숫자, 통화 KRW 등)
- workHoursPerYear: 시간당 계산용 기준 (기본값 제공, 수정 가능)
- hourlyLevelCost: 자동 계산 결과(읽기 전용)
- isActive: 사용 여부(비활성 시 신규 배정 불가)

### **2-3. 자동 계산 규칙(시간당 레벨비용)**

- hourlyLevelCost = avgAnnualSalary / workHoursPerYear
- workHoursPerYear 기본값 예시(선택):
    - **2080**(주 40h × 52주)
    - 또는 회사 정책에 맞춘 값(예: 휴일/연차 반영한 2000 등)
- 중요한 건 “정책값을 고정”해서 **전체 비용 산정의 일관성** 확보.

### **2-4. 레벨 개수/추가/수정 정책**

- 레벨은 **무제한 추가 가능**.
- 평균연봉 수정 시:
    - **향후 계산에만 적용**(권장) vs **전체 과거 데이터 재계산**(비권장, 회계 흔들림)
    - 현실적으로는 “유효기간(Effective Date)” 개념을 두는 게 가장 깔끔함:
        - 레벨 단가 버전: effectiveFrom을 두고, 프로젝트 원가 계산 시 해당 날짜에 유효한 단가를 사용.

### **2-5. 필수 검증/예외**

- regionGroup + levelName 중복 방지(또는 중복 허용하되 식별자 별도 필요)
- avgAnnualSalary 0 이하 입력 불가
- 레벨 삭제는 원칙적으로 금지(과거 데이터 참조 가능성) → **비활성(isActive=false)**로 처리

## **3) 직원 배정 규칙(실제 연봉 입력 없이 “레벨만”)**

### **3-1. 직원 데이터에서 관리하는 값**

- 직원(Employee)은 다음만 가진다:
    - regionGroup (KOR/SEA)
    - levelId (해당 regionGroup의 레벨 중 선택)
    - hourlyLevelCost는 **레벨에서 자동 참조**(직원에 별도 저장하지 않아도 됨)

### **3-2. 레벨 자동 배정(옵션)**

너가 말한 “연봉 입력 시 자동 레벨 배정”은 현재 정책(실제비용 노출X)과 충돌하니까, 아래 중 하나로 정리하는 게 좋아:

**안 A(권장): 레벨을 수동 선택**

- 직원 추가/수정 시 regionGroup 선택 → 해당 그룹의 레벨 리스트에서 선택
- 가장 단순하고 실제비용과 완전히 분리됨

**안 B(자동 배정 유지): ‘연봉’이 아니라 ‘밴드/레인지’ 입력**

- 실제 연봉 숫자 대신, 내부적으로만 쓰는 **밴드 코드**(예: “Band 5”) 입력 → 레벨 매핑
- 어쨌든 “실제 연봉 숫자”는 시스템에 안 들어오게 함

## **4) 프로젝트 비용 산정 방식**

### **4-1. 모든 프로젝트 원가 산정의 기준**

- 프로젝트 원가/수익성 계산 시 **레벨 비용만 사용**:
    - 예정 원가(Planned Cost): Σ(예정 투입시간 × hourlyLevelCost)
    - 실제 원가(Actual Cost): Σ(실제 투입시간 × hourlyLevelCost)
- “실제비용(연봉 기반)”이라는 개념 자체가 시스템에 없음.

### **4-2. 레벨 비용이 바뀔 때의 처리**

- 비용이 뒤바뀌지 않게 하려면 원가 계산은 아래 중 택1:
    1. **단가 버전/유효기간 기반**(권장): 작업일/투입일 기준으로 해당 시점의 단가 적용
    2. **프로젝트 시작일 기준 고정**: 프로젝트 시작 시점 단가로 프로젝트 전체 고정
- (권장순) 1 > 2 > “항상 최신단가로 과거까지 재계산”(비권장)

---

## **5) 직원 관리 페이지 노출 정책**

### **5-1. 컬럼 구성(변경)**

기존의

- 연봉
- 시간당비용(실제)

을 제거하고, 아래만 노출:

- Region (KOR/SEA)
- Level (레벨명)
- Hourly Level Cost (시간당 레벨비용)
- Role / Skill / 고용형태(원하면 유지)

### **5-2. 상세/편집 UX**

- 직원 행 클릭 → Drawer/Modal
    - 수정 가능: regionGroup, level, role, skill, 고용형태
    - 수정 불가(또는 없음): 연봉, 실제 시간당비용
- 레벨 변경 시 즉시 “시간당 레벨비용” 표시가 바뀜(참조값)

---

## **6) 화면/플로우 요약**

1. Settings에서 KOR/SEA별 레벨 생성
2. 레벨마다 평균연봉 입력 → 시간당 레벨비용 자동 계산
3. 직원 추가 시 KOR/SEA 선택 + 레벨 선택
4. 모든 프로젝트/워크포스/파이낸스 원가 계산은 레벨단가 기반
5. 직원관리 테이블은 레벨/레벨단가만 노출 (실제비용 관련 컬럼 0)
