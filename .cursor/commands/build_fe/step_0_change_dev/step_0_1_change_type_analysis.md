# Step 0-1: 변경 유형 분석 (Change Type Analysis)

## Overview
입력으로 주어진 “변경된 기획/요구사항”을 기존 요구사항 기준으로 분해해 **추가/삭제/수정/영향도 변화**로 분류하고, 이후 영향 범위 분석과 실행 계획의 기준을 만듭니다.

## Steps
1. 변경 요구사항 원문을 그대로 보존할 준비를 합니다(이 단계에서는 아직 문서 파일을 만들지 않아도 됨).
2. 변경 항목을 가능한 한 작게 쪼개 “원자적 요구사항” 리스트로 만듭니다.
3. 각 항목을 아래 4가지로 분류합니다.
   - Added: 새로 추가된 요구사항
   - Removed: 삭제된 요구사항
   - Modified: 기존 요구사항의 내용/동작/조건이 바뀜
   - Impact-shift: 요구는 동일하지만 우선순위/범위/리스크/제약이 바뀜
4. 각 항목에 최소 정보를 붙입니다.
   - 변경 요약(1줄)
   - 근거/원문 인용(짧게)
   - 영향 모듈 후보(Fxx/US-xxx/Route 등, 아직 가설이어도 됨)
5. Step 0-2(영향 범위 분석)로 넘길 “변경 항목 목록”을 확정합니다.
6. (변경사항이 생긴 경우) `git status` 확인 → `git add` → `git commit`

## Checklist (선택)
- [ ] 변경 항목이 너무 뭉쳐 있지 않은가? (한 항목=한 변화)
- [ ] Added/Removed/Modified/Impact-shift 중 하나로 모두 분류되는가?
- [ ] 각 항목에 원문 근거가 남아 있는가?
- [ ] (변경사항이 생긴 경우) 커밋까지 완료했는가?
