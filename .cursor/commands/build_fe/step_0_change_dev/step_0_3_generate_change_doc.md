# Step 0-3: 변경 문서 생성 (docs/changes/YYYY-MM-DD.md) — UI Only

## Overview
> **⚠️ UI 변경만 다룹니다.** API/DB/Migration은 이 커맨드 범위가 아닙니다.

기획 변경의 **원문 아카이브 + 분석 결과 + 실행 계획**을 한 파일(`docs/changes/YYYY-MM-DD.md`)에 고정 포맷으로 남겨, 이후 개발/리뷰/회귀검증의 기준 문서로 사용합니다.

## Steps
1. 오늘 날짜(또는 변경 요청이 확정된 날짜)로 `docs/changes/YYYY-MM-DD.md`를 생성합니다.
2. 아래 목차를 따라 문서를 채웁니다.
   - 입력 원문(요약 + 원문)
   - 변경 유형 분석(Added/Removed/Modified/Impact-shift)
   - 영향 범위 분석(화면/컴포넌트/상태/스타일)
   - 영향 모듈 리스트(우선순위/의존성)
   - 실행 계획(모듈별 Step 8 루프 적용 순서)
   - 리스크/TBD(결정 필요 시 등록/해결 흐름)
   - 완료 정의(검증/회귀 범위, Done 조건)
3. `docs/dev_plan.md`가 존재한다면, 변경으로 인해 우선순위/의존성이 바뀐 경우 간단히 갱신합니다(선택).
4. 다음 단계(Step 0-4)에서 바로 실행할 수 있도록 "현재 개발 대상 모듈"을 문서에 표시합니다.
5. (변경사항이 생긴 경우) `git status` 확인 → `git add` → `git commit`

## Checklist
- [ ] 원문이 누락 없이 아카이브되어 있는가?
- [ ] 영향 모듈/우선순위/의존성이 명확한가?
- [ ] Step 8을 모듈별로 바로 실행 가능한 수준인가?
- [ ] (변경사항이 생긴 경우) 커밋까지 완료했는가?

## Output (예시)
생성해야 하는 문서: `docs/changes/YYYY-MM-DD.md`

```markdown
# Change Log: YYYY-MM-DD

## 0) 입력 원문
### 요약
- (1~5줄로 변경 핵심)

### 원문
- (원문 그대로)

---

## 1) 변경 유형 분석
### Added
- CHG-001: ... (근거: "...")

### Removed
- CHG-002: ...

### Modified
- CHG-003: As-is → To-be ...

### Impact-shift
- CHG-004: 요구는 동일, 범위/우선순위 변경 ...

---

## 2) 영향 범위 분석 (UI Only)
| 영역 | 필요 여부 | 변경 내용(요약) | 영향 모듈 | 산출물/비고 |
|---|:---:|---|---|---|
| 화면(Route) | ✅ | ... | F03, Route:/... | 신규 페이지/레이아웃 |
| 컴포넌트 | ✅ | ... | F03 | 컴포넌트 수정/추가 |
| 상태/Interaction | ⚠️ | ... | F03 | 훅/store 변경 |
| 스타일/UX | ❌ | - | - | - |

---

## 3) 영향 모듈 리스트 & 우선순위
| 모듈 | 설명 | 우선순위 | 상태 |
|---|---|---|---|
| F03 | ... | High | TODO |

### 의존성(요약)
| From | To | 이유 |
|---|---|---|
| F01 | F03 | 권한/세션 선행 |

---

## 4) 실행 계획 (Step 8 적용)
- 대상 모듈: F03
- 수행: Step 8-1 → 8-2 → 8-3 → 8-4 → 8-5 (모듈별 반복)

---

## 5) 리스크 / TBD
- TBD-001: ... (결정 필요)

---

## 6) 완료 정의 (Definition of Done)
- [ ] 변경 항목 CHG-001~가 모두 반영됨
- [ ] 핵심 플로우 회귀 테스트 통과
- [ ] 린트/타입 에러 없음
```
