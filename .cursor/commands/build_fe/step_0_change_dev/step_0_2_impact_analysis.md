# Step 0-2: 영향 범위 분석 (Impact Analysis) — UI Only

## Overview
> **⚠️ UI 변경만 분석합니다.** API/DB/Migration은 이 커맨드 범위가 아닙니다.

Step 0-1의 변경 항목이 실제로 **UI(화면/컴포넌트/상태)**에 어떤 영향을 주는지 분석하여, 실행 계획(우선순위/의존성/리스크)을 만들 수 있게 합니다.

## Steps
1. Step 0-1의 변경 항목(Added/Removed/Modified/Impact-shift)을 입력으로 받습니다.
2. 아래 항목별로 "필요 여부/변경 내용/영향 모듈/산출물"을 채웁니다.
   1) **화면(Route) 영향**
      - 신규 화면 필요?
      - 기존 화면 레이아웃/플로우 변경?
   2) **컴포넌트 영향**
      - 기존 컴포넌트 수정?
      - 신규 컴포넌트 필요?
   3) **상태/Interaction 영향**
      - 상태 관리(훅/store) 변경?
      - 사용자 interaction 변경?
   4) **스타일/UX 영향**
      - 디자인/스타일 변경?
      - 접근성/반응형 영향?
3. 분석을 바탕으로 "영향 모듈 리스트(Fxx/US-xxx/Route)"를 확정합니다.
4. 모듈 간 의존성을 간단히 정리합니다(From → To → 이유).
5. 미결정/불명확(TBD)이 있으면 목록으로 분리해 기록합니다(다음 단계에서 변경 문서에 포함).
6. Step 0-3(변경 문서 생성)로 넘길 요약을 준비합니다.
7. (변경사항이 생긴 경우) `git status` 확인 → `git add` → `git commit`

## Checklist (선택)
- [ ] 화면/컴포넌트/상태/스타일 영향이 모두 분석되었는가?
- [ ] 영향 모듈이 "가설"이 아니라 실행 가능한 수준으로 수렴했는가?
- [ ] TBD가 분리되어 있고, 해결 없이는 진행 불가한 항목이 표시되었는가?
- [ ] (변경사항이 생긴 경우) 커밋까지 완료했는가?
