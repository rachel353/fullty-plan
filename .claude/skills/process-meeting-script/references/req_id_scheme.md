# REQ ID Scheme

## Format
```
REQ-XYY
```
- **X**: 미팅 순서 번호 (1=첫 번째 미팅, 2=두 번째 미팅, ...)
- **YY**: 카테고리 내 순차 번호 (01-99)

## Meeting Number Determination

1. 클라이언트 폴더 내 `meeting_scripts/` 하위 폴더 목록 확인
2. 날짜 기준 정렬 (MM.DD 형식)
3. 현재 미팅의 순서 번호 결정

### Example
```
meeting_scripts/
├── 11.01/  → 첫 번째 미팅 (X=1)
├── 11.06/  → 두 번째 미팅 (X=2)
└── 11.15/  → 세 번째 미팅 (X=3)
```

## ID Ranges by Category

| Meeting | 핵심 기능 | 비기능 | 통합 | UI/UX | 비즈니스 |
|---------|----------|--------|------|-------|---------|
| 1st | REQ-101~110 | REQ-111~120 | REQ-121~130 | REQ-131~140 | REQ-141~150 |
| 2nd | REQ-201~210 | REQ-211~220 | REQ-221~230 | REQ-231~240 | REQ-241~250 |
| 3rd | REQ-301~310 | REQ-311~320 | REQ-321~330 | REQ-331~340 | REQ-341~350 |

## Rules

1. **중복 방지**: 동일 미팅 내에서 REQ ID 중복 불가
2. **연속 할당**: 각 카테고리 내에서 순차적으로 할당
3. **미팅 번호 고정**: 미팅 날짜가 바뀌어도 순서 번호는 정렬 기준 유지
4. **이전 ID 참조**: 이전 미팅의 requirements.md 확인하여 중복 방지
