# Fill Test Cases

Fill TCs into skeleton JSON and verify coverage completeness.

## Prerequisites

- Skeleton JSON exists (generated by `/create-testcase`)
- Conceptual model document available for Domain Rules
- Test credentials document available for preconditions

## Workflow (3 Steps)

### Step 1: Fill Test Cases per AC

For each AC with empty `testCases` array, create TCs following:

**TC ID format:** `TC-<US>-<AC>-<NN>` (e.g., `TC-US-009-AC-2-01`)

**Minimum TC set per AC:**
1. Positive (정상) - 1개 필수
2. Negative (실패) - 1개 권장
3. Edge/Time/UX - 해당시 추가

See [references/tc-patterns.md](references/tc-patterns.md) for type definitions and examples.

### Step 2: Enrich from Conceptual Model

Read conceptual model and ensure:

1. **Domain Rules (DR-xxx)**: Each rule has at least 1 TC
   - Add DR-xxx to TC's `tags` array
2. **Data Flows**: Add E2E TC for flows spanning multiple ACs

### Step 3: Coverage Check

Run coverage verification:

```bash
python3 .claude/skills/fill-testcases/scripts/coverage_check.py \
  --input <path-to-testcases.json>
```

**Coverage checklist:**
- [ ] All AC have `testCases.length >= 1`
- [ ] All DR-xxx have at least 1 TC
- [ ] All CI-xxx have at least 1 TC
- [ ] Critical routes have smoke + access TC

## Output

Update testcases JSON with:
- Filled `testCases` arrays in each AC
- Updated `coverage` section with TC IDs

## Quick Reference

| Field | Description |
|-------|-------------|
| `id` | TC-US-XXX-AC-N-NN format |
| `title` | "[타입]: 설명" format |
| `type` | positive, negative, edge, smoke, access |
| `precondition` | 테스트 전 필요 상태 |
| `steps` | 번호 붙은 단계별 행동 |
| `expected` | 검증 가능한 결과 |
| `status` | todo (초기값) |
| `tags` | 관련 DR-xxx, CI-xxx |
